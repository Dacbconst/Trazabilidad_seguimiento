Al diseñar una feature, pensar de forma lógica/analítica en los casos de uso reales que la rodean — no solo implementar el pedido literal, sino preguntarse qué pasa antes, durante y después de esa acción en la operación real del negocio.

**Por qué:** El usuario señaló que una feature "casi completa" en realidad tenía huecos operativos obvios una vez pensados con cuidado: (1) se podía marcar una visita como "reagendada" pero la fecha era de solo lectura, así que en la práctica nunca se podía reagendar para otro día; (2) se pidió "eliminar" un agendamiento, y la tabla ya tenía una columna `activar` pensada exactamente para borrado lógico — usar `estado_agenda` para eso habría confundido un estado de negocio real (cancelado) con una corrección de error; (3) marcar una visita como "vencida" en una lista no sirve si al abrir el detalle no hay ningún aviso que empuje a corregirla.

**Cómo aplicar:** Para cada acción nueva (editar, eliminar, cancelar, marcar estado), recorrer mentalmente el flujo completo: ¿quién la usa y cuándo?, ¿qué pasa con los datos relacionados (calendario, listas, mapa, leyenda)?, ¿la UI realmente permite completar la intención del usuario o solo la parte visible del pedido? Si la tabla ya tiene una columna pensada para algo (como `activar` para borrado lógico), usarla en vez de sobrecargar otra columna con un significado distinto. Ver también [[feedback_pensamiento_critico_ux]].

**Dos errores concretos que ya cometí y no debo repetir** (en [agendamientos.php](Proyectos/Pintuco/components/agendamientos.php) / [get_agenda.php](Proyectos/Pintuco/getters/get_agenda.php)):
1. *Asumir el tipo/valor "vacío" de una columna sin verificarlo.* Agregué un filtro SQL `activar != 0` asumiendo que era numérico/NULL por defecto. La columna real es `varchar(2) NOT NULL DEFAULT 'SI'` con valores `'SI'/'NO'` — confirmado consultando la tabla en vivo con un getter PHP temporal (`SHOW COLUMNS` + `GROUP BY activar`) llamado directo por URL (la app corre en Azure App Service, así que un `WebFetch` a la URL pública del getter es exactamente como pegarle con Postman, no requiere nada instalado). `'SI' != 0` se coacciona a `0 != 0` → falso → excluí TODAS las filas (regresión real: "Agendas pendientes" pasó a mostrar 0). Lección: ante cualquier duda sobre el tipo/valores reales de una columna en este proyecto, escribir un getter de diagnóstico temporal y pedirle al usuario la URL pública (o el comando para correrlo) en vez de adivinar — es rápido y queda confirmado con datos reales, no con suposiciones. `activar` en este proyecto es `'SI'`/`'NO'`, no `1`/`0`.
2. *Colocar un control propio (botón `position:absolute`) encima de los controles nativos de una librería de terceros* (el toggle para colapsar el mini-calendario quedó flotando justo sobre el botón "next" de FullCalendar, pisándose visualmente). Antes de usar `position:absolute` sobre un widget de terceros (FullCalendar, Leaflet, etc.), revisar dónde ese widget ya pone sus propios controles (toolbar, botones) y, si hay riesgo de choque, sacar el control propio a una barra/elemento separado en vez de superponerlo.
3. *Calcular el mismo "estado visible" con dos lógicas distintas en dos lugares de la UI, y que diverjan.* La lista de "Agendas pendientes" decidía si una visita estaba completa mirando directamente `r.hora`/`r.tecnico`; el badge del modal decidía el estado con `normalizarEstado(estado_agenda)`, que asumía "agendada" para cualquier cosa que no fuera cancelada/reagendada — sin mirar si en verdad tenía hora/técnico. Resultado: una card podía decir "SIN AGENDAR" en la lista y "Agendada" (verde) al abrirla. El estado real de una entidad debe derivarse de **una sola función**, basada en las condiciones reales de sus datos (¿tiene hora y técnico?), no de una columna de estado aislada ni de una señal incidental de la UI (como "este campo es editable, entonces debe ser que no está agendada"). Esa única función (`estadoVisual()`) se usa ahora en badge, lista de pendientes, color del evento en el calendario y popup del mapa — si se agrega una nueva vista que muestre el estado, debe consumir esa misma función, nunca reinventar la condición.
